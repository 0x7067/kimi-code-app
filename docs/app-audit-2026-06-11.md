# Kimi Code App Audit - 2026-06-11

## Scope

Requested audit: open the app, exercise every feature, and document functional or UX/UI issues.

What was verified directly:
- `cargo tauri dev` launches the backend binary, but the dev webview stays on the Dioxus build/loading screen.
- The bundled macOS app at `target/release/bundle/macos/kimi-code-app.app` renders the main UI.
- `cargo test --workspace` passes: 149 tests, 0 failures.
- `cargo check --target wasm32-unknown-unknown` passes.
- `cargo check --workspace` passes.

Interaction limitation:
- macOS Accessibility/System Events access is unavailable in this Codex session, and synthetic Quartz clicks did not change app state. I could not honestly claim full manual click-through coverage of every panel. Issues below are split into verified runtime issues and source-backed product issues.

## Functional Issues

### P0 - Dev launch gets stuck on Dioxus loader

Evidence:
- `cargo tauri dev` starts `dx serve --port 1420`, compiles the Tauri binary, and runs `target/debug/kimi-code-app`.
- Fresh app window remains on: `One sec! We're building your app now. Starting the build... 0%`.
- `curl -I http://localhost:1420/` and generated `/wasm/kimi-code-app-ui.js` / `.wasm` assets return `200 OK`.
- Reloading the webview with Command-R did not recover it.
- Bundled app renders correctly, so this is specific to the dev server/webview path.

Impact:
- The documented development workflow is unreliable. UI/visual changes cannot be smoke-tested with the required `cargo tauri dev` path.

Likely fix area:
- Dioxus dev bootstrap or Tauri dev URL integration. Check webview console/runtime errors around `/./wasm/kimi-code-app-ui_bg.wasm`, Dioxus loader state, and `dx serve` hot-reload/bootstrap scripts.

### P1 - Multiple app instances can stack with identical window title and bounds

Evidence:
- WindowServer showed multiple `kimi-code-app` windows with title `Kimi Code`, same bounds, different PIDs after opening debug and bundled builds.
- Some windows captured as stale Dioxus loader while another rendered the real app.

Impact:
- Developers can end up testing a stale window and misread the actual app state.

Likely fix area:
- App single-instance handling, dev/release process cleanup, or clearer dev window title/version marker.

### P1 - Automation editor saves invalid jobs without validation

Evidence:
- `AutomationEditor` saves name, cron, cwd, prompt directly into settings.
- Backend `cron_due_between` silently ignores invalid cron expressions.
- Empty prompt/cwd/name are not blocked in the editor.

Impact:
- Users can create automations that never run or fail later with little feedback.

Likely fix area:
- Validate cron, name, prompt, and cwd before saving. Surface inline errors.

### P1 - Automation Run Now does not refresh history or show success feedback

Evidence:
- `Run now` invokes `run_automation_now`; success path is `Ok(_) => {}`.
- Execution history only updates after manual `Refresh history`.

Impact:
- A real call can succeed while the UI appears unchanged.

Likely fix area:
- Refresh run history after success and show status/toast.

### P1 - Multi-agent "Run all" has race-prone state updates

Evidence:
- `Run all` spawns one `run_multi_agent_task` per task concurrently.
- Each completion does `current_run.set(v)` with the returned run snapshot from that single task call.

Impact:
- Last finisher can overwrite visible state with a snapshot that may not include other concurrent updates depending on timing.

Likely fix area:
- Poll/refetch the run after all tasks settle, or have a single backend command orchestrate all tasks and return final run state.

### P2 - Browser preview has no load/error feedback

Evidence:
- Browser pane defaults to `http://localhost:3000`, renders an iframe, and provides Load/Refresh/device/share controls.
- There is no visible loading state, failed-load state, URL validation, or iframe error reporting.

Impact:
- Blank/blocked pages look like broken app chrome rather than a target-site failure.

Likely fix area:
- Add loading indicator, last-loaded URL, error state, and Open External fallback.

### P2 - Automation history displays raw epoch timestamps

Evidence:
- `RunRow` renders `startedAt` directly as `{started}`.

Impact:
- History is hard to scan and feels unfinished.

Likely fix area:
- Use existing relative/date formatting helpers or localized date/time.

### P2 - Terminal pane is a plain text renderer, not terminal emulation

Evidence:
- `terminal_pane.rs` explicitly says "simple text renderer - no full xterm emulation".

Impact:
- Many commands that use ANSI control sequences, progress bars, alternate screen, or rich TUI output will render poorly.

Likely fix area:
- Either integrate a real terminal renderer or constrain UX copy to "command output" instead of "Terminal".

## UX/UI Issues

### P1 - Empty state blocks primary action clarity

Evidence:
- Main empty state says "Pick a project and start a new session, or resume one from the sidebar."
- Composer is disabled with placeholder "Start a session first."
- There is no central primary button to start a new session.

Impact:
- The first screen tells users what to do but does not offer the action where their attention is focused.

Likely fix area:
- Add primary "New chat" / "Choose project" action in the empty state.

### P2 - Topbar is visually crowded and text-heavy

Evidence:
- Search, Checkpoints, Diff, Terminal, Memory, Browser, Multi-agent, Automations, Settings all compete as full text buttons.

Impact:
- At 1280px width, the topbar feels like a control dump and leaves little hierarchy between frequent and advanced actions.

Likely fix area:
- Use icons with tooltips for secondary panels, group advanced tools, or move panel toggles into a command/menu area.

### P2 - Sidebar session titles are heavily truncated

Evidence:
- Existing sessions show repeated `Reply exactly: FINAL_LI...` rows.

Impact:
- Session list is hard to distinguish when titles share prefixes.

Likely fix area:
- Use two-line title wrapping, better semantic title generation, or include last prompt/result summary.

### P2 - Status semantics are inconsistent

Evidence:
- Topbar shows green dot + `Kimi Code CLI 0.14.0`.
- Bottom status shows blue dot + same agent/version + `Ready`.

Impact:
- Duplicate status indicators add noise and use different colors for apparently related state.

Likely fix area:
- Keep connection identity in one place; use the bottom bar for detailed runtime status.
